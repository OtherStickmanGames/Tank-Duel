# Tank Duel — план разработки (рабочее название)

> Прототип: лоуполи-танки, вид сверху, стилизованное 3D, **Unity 6 (6000.x LTS) + URP 17**.
> Петля: **60 сек фарма на своей половине → стена падает → дуэль → награда**.
> Платформы: **Web** (Yandex Games / Playhop) + Android. Сеть — своя (мастер-сервер + WebSocket), боты-заполнители при пустом матчмейкинге.

---

## Как пользоваться этим планом (для ИИ и для себя)

- **Одна задача = одна сессия с ИИ.** Не проси «сделай этап 1», проси «сделай задачу 1.2». Скидывай ИИ этот файл + уже написанные скрипты задачи.
- У каждой задачи есть **Готово, когда** — критерий приёмки. Не переходи дальше, пока не выполнен.
- Задачи внутри этапа идут по порядку — каждая опирается на предыдущую.
- После каждой задачи — коммит. Название коммита = номер и название задачи.

### Unity AI vs. я — кто что делает

В Unity 6 встроен свой ИИ (Assistant + генераторы ассетов). Разделение труда:

| Задача | Кому |
|---|---|
| Генерация текстур, материалов, спрайтов, иконок | **Unity AI** — прямо в редакторе, сразу в проект |
| «Где эта настройка в Unity 6», навигация по редактору | **Unity AI** — знает конкретную версию редактора |
| Вопросы по своему же проекту (что делает этот компонент) | **Unity AI** — видит сцены и ассеты |
| Архитектура, сетевой код, геймплейная логика | **я** — держу весь план и связи между системами в голове |
| Многофайловый рефакторинг, отладка сложных багов | **я** |

Практическое правило: **ассеты и редактор — Unity AI, код и решения — я.** Мой потолок знаний по Unity — начало 2026, так что по свежим API Unity 6 лучше перепроверять через Unity AI или доки, а не верить мне на слово.

### Какой моделью что делать

| Модель | Когда брать | Задачи в плане |
|---|---|---|
| **Fable 5** | Архитектура, сетевой код, всё что «сложно и дорого переделывать» | помечено `[Fable]` |
| **Opus 4.8** | Основной геймплейный код, ИИ ботов, интеграции с площадками | помечено `[Opus]` |
| **Sonnet 5** | Чётко описанные задачи по шаблону: UI, спавнеры, конфиги, рутина | помечено `[Sonnet]` |

Правило: если задача **создаёт структуру**, на которую потом всё вешается — Fable. Если **наполняет готовую структуру** — Opus. Если **повторяет уже существующий образец** (второй экран UI по образцу первого и т.п.) — Sonnet.

## Принятые решения (не пересматривать без причины)

1. **Движок:** Unity 6 (6000.x LTS), URP 17. Никаких смен движка.
2. **Сеть:** настоящий мультиплеер, **NGO 2.x** (идёт с Unity 6), транспорт — **официальный UnityTransport с включённым WebSocket**. Матчмейкинг — свой мастер-сервер на имеющейся VPS.
3. **Код из World-of-Mine не переносим — ни строчки.** WoM это воксельный мир (чанки, строительство, симуляция), его сетевой код к танковой арене отношения не имеет, плюс он написан под NGO 1.12.2 с другими идиомами. Пишем с нуля под 2.x.
   **Из WoM берём только инфраструктуру и опыт:**
   - `reverse-proxy.conf` — рабочий nginx для проксирования WebSocket через HTTPS (заголовки Upgrade/Connection, длинные таймауты, `tcp_nodelay`). Место, где легко потерять полдня; у тебя уже решено.
   - Схема с Certbot — на Yandex/Playhop страница отдаётся по HTTPS, значит браузеру нужен `wss://`, значит сертификат обязателен.
   - Общий опыт эксплуатации мастер-сервера на этой VPS.
4. **Боты-заполнители:** если за N секунд не нашёлся живой соперник — матч против бота. Игрок и бот управляются через один интерфейс ввода, для остального кода разницы нет.
5. **Билд танка — данные, а не поля в компонентах.** Сериализуемый `TankBuild`, из которого танк собирается. Это фундамент и для ботов, и для сети, и для карточек.
6. **Первая версия — только PvP-петля.** Карточки, комбинирование, кооп — после плейтеста.
7. **Не привязывать ценные награды к победе в PvP**, пока результат матча не валидируется сервером (хост может мухлевать).
8. **Все необходимые объекты сцены генерируются Editor-скриптами.** Арена, стена, спавн-поинты, менеджеры, UI-каркас — всё создаётся кодом из `Assets/Scripts/Editor/` через пункты меню (`Tank Duel → Build Arena` и т.п.), а не руками в иерархии. Зачем: ИИ пишет код с VPS и не имеет доступа к редактору — так любая задача «собери сцену» превращается в «нажми пункт меню». Сцена воспроизводима: снёс объект — пересоздал заново одним кликом. Генераторы должны быть идемпотентными: повторный запуск не плодит дубли, а пересоздаёт/обновляет существующее. Руками в сцене остаётся только то, что требует художественного глаза (свет, камера-тюнинг, декор).

## Этап 0 — Каркас проекта

> **Статус на 21.07.2026:** 0.1 ✅ (Unity 6000.5.4f1, NGO 2.13.0, URP 17.5.0, Brotli/Stripping High выставлены, шаблон вычищен). 0.2 ✅ настройки (NetworkManager + UnityTransport, Use WebSockets вкл, encryption выкл — TLS терминирует nginx).
> **Решение по тестированию:** сетевые проверки в разработке гоняем **Standalone PC билдами через localhost**. Проверку «Web-билд коннектится по WebSocket» отложили — ⚠️ провести не позже начала этапа 4, это главный технический риск всей затеи.

### 0.1 Создать проект и настроить под Web `[Sonnet]`
Пустой проект **Unity 6 (6000.x LTS)**, URP-шаблон. Сразу:
- Input System (new). **Newtonsoft JSON не ставим** — для `TankBuild` хватает встроенного `JsonUtility`, а лишняя зависимость это лишние килобайты в Web-билде. Добавим, только если упрёмся в словари/полиморфизм.
- Платформа **Web** (в Unity 6 «WebGL» переименован в «Web»): Brotli, Managed Stripping High, Decompression Fallback off
- Графика: **WebGL 2 основным**. WebGPU в Unity 6 есть, но пока экспериментальный — в план не берём, можно потрогать на этапе 6.1 как возможный буст
- Отключить неиспользуемые модули (VR, Terrain, Cloth, Video и т.п.)
- Папки: `Scripts/{Core,Tank,AI,UI,Net,Data}`, `Prefabs`, `ScriptableObjects`
- Git-репозиторий, Unity `.gitignore`, **приватный** с первого коммита. Никаких keystore/паролей в репе.

**Готово, когда:** пустая сцена собирается в Web ≤ 10 МБ и открывается в браузере, в том числе на телефоне (Unity 6 официально поддерживает мобильные браузеры — это как раз наша аудитория на Yandex/Playhop).

### 0.2 Поднять NGO 2.x с WebSocket-транспортом `[Opus]`
1. Поставить NGO 2.x (Package Manager, идёт с Unity 6), поднять NetworkManager + UnityTransport.
2. Включить в UnityTransport **WebSockets**, проверить связку «Web-билд ↔ хост» локально.
3. Сетевую логику пока не писать — только чтобы поднималось и коннектилось.

Если официальная поддержка WS почему-то не заведётся — **это стоп-сигнал, разбираемся отдельной сессией**, а не тащим кастомный транспорт из WoM «чтобы работало». Кастомный код = вечная поддержка чужого websocket-sharp.

**Готово, когда:** NetworkManager поднимается в режиме host, Web-билд к нему коннектится, в консоли чисто.

## Этап 1 — Ядро: данные и стейт-машина

> **Статус:** 1.1 ✅, 1.2 ✅ (приняты 21.07.2026, TankBuildSelfCheck удалён). 1.3 — код в репозитории, приёмка не отмечена явно (пользователь погнал дальше). 1.4 — код в репозитории.
> Приёмка 1.4: `Tank Duel → Build Arena` → Play → едешь к стене, упираешься физически; стреляешь в стену — снаряд гаснет об неё; ставишь `farmDuration = 3` на MatchController для быстрого теста → после Warmup+Farm стена ~1.5 сек опускается (WallDrop) → в Duel можно проехать и прострелить туда, где раньше была стена. Этап 1 «Ядро» этим закрывается.
> **Решение: никаких юнит-тестов в прототипе.** Всё проверяется глазами в Play Mode. Вопрос тестов можно поднять после плейтест-чекпоинта, если ядро выживет — и только для слоя данных.

### 1.1 `TankBuild` — структура данных билда `[Fable]`
Сериализуемый класс: корпус, орудие, уровни прокачки (урон/скорострельность/скорость/HP), список перков (пока пустой enum-задел). Методы: `ToJson/FromJson`, расчёт итоговых статов (`TankStats ComputeStats()`).
Плюс `ScriptableObject`-конфиги: `TankConfig`, `UpgradeConfig` (цены и приросты по уровням — всё числовое в конфигах, ничего в коде).

**Готово, когда:** временный скрипт в сцене (создать билд → апгрейдить → в JSON и обратно → вывести статы в консоль) показывает ожидаемые числа. Скрипт после проверки удалить.

### 1.2 `MatchController` — стейт-машина матча `[Fable]`
Состояния: `Warmup → Farm(60s) → WallDrop → Duel → Result`. События на входы/выходы состояний (`OnFarmStarted`, `OnDuelStarted`, ...). Таймер фарма. Никакой логики боя внутри — только оркестрация. Должна одинаково работать локально и (в будущем) на хосте по сети — поэтому вся смена состояний в одном месте, остальные подписываются.

**Готово, когда:** в тестовой сцене состояния переключаются по таймеру, события летят, в консоли видна полная петля матча.

### 1.3 `ITankInputSource` + `TankController` `[Fable]`
Интерфейс ввода: направление движения, направление прицела, огонь. Реализации: `PlayerInputSource` (Input System: WASD/стик + мышь/второй стик), позже `BotInputSource` и `NetworkInputSource`.
`TankController`: движение (свободное, 8 направлений, раздельное прицеливание башни), стрельба снарядами, HP, смерть. Все числа — из `TankStats` (задача 1.1). Один и тот же префаб для игрока, бота, сетевого оппонента.

**Готово, когда:** в сцене катается танк с капсулами-мишенями, стреляет, мишени умирают; смена чисел в конфиге меняет поведение без правки кода.

### 1.4 Арена и стена `[Sonnet]`
Editor-скрипт `Tank Duel → Build Arena` (решение №8): генерирует две половины, стену посередине (поднята/опущена, анимация опускания по событию `WallDrop`), спавн-поинты, камеру сверху с лёгким наклоном — обе половины в кадре. Грейбокс из примитивов.

**Готово, когда:** по событию из MatchController стена опускается, до этого физически непроходима и непрострелима.

## Этап 2 — Фаза фарма

### 2.1 `Spawner` — боты и разрушаемые объекты `[Opus]`
Спавн волн фарм-ботов и разрушаемых объектов в пределах заданной половины арены. Конфиг волн в ScriptableObject (что, сколько, когда). Пул объектов сразу (WebGL не любит аллокации).

**Готово, когда:** во время `Farm` на половине игрока появляются боты/объекты по конфигу, после `Farm` спавн останавливается, всё живое зачищается.

### 2.2 Фарм-боты (тупой ИИ) `[Sonnet]`
`BotInputSource` для фарм-ботов: бродят, при виде игрока едут на него и стреляют. Намеренно простые — их роль быть кормом. Используют тот же `TankController` (или упрощённый `DestructibleUnit` для не-танков).
*Опция Unity 6:* пакет **Unity Behavior** (визуальные деревья поведения) — для тупых фарм-ботов избыточен, обычный код проще и дешевле по производительности. Рассмотреть на задаче 3.1, где логика сложнее.

**Готово, когда:** боты умирают, дают очки, изредка «кусаются». Игрок за 60 сек успевает набить 10–20 целей.

### 2.3 `ScoreSystem` + прокачка в матче `[Sonnet]`
Очки за убийства/разрушения. UI-панель прокачки: 4 кнопки (урон/скорострельность/скорость/HP), цена растёт по `UpgradeConfig`, апгрейд применяется через `TankBuild` немедленно. Прокачка доступна только в фазе `Farm`.

**Готово, когда:** полный цикл: убил → получил очки → купил апгрейд → танк ощутимо изменился. Панель блокируется вне фазы фарма.

### 2.4 Зеркальный фарм оппонента `[Opus]`
На верхней половине то же самое делает бот-«игрок»: фармит, тратит очки на апгрейды (простая эвристика: копит и покупает случайно-взвешенно). Его прогресс виден игроку (счёт вверху экрана) — это создаёт напряжение «он тоже качается».

**Готово, когда:** к моменту падения стены у бота осмысленный билд, сопоставимый с игроком ±30%.

## Этап 3 — Дуэль и ЧЕКПОИНТ

### 3.1 Дуэльный ИИ `[Opus]`
`BotInputSource` режим «дуэль»: сближается/кайтит в зависимости от своего билда против билда игрока, стрейфит, стреляет с упреждением (с настраиваемой ошибкой — ручка сложности). Не идеальный — «средний игрок».
*Опция Unity 6:* здесь **Unity Behavior** уже оправдан — поведение ветвистое, а визуальный граф удобно крутить без перекомпиляции. Решение принять в начале задачи: граф даёт скорость итераций, чистый код даёт контроль и меньше зависимостей. Для 1–2 ботов на сцене производительность роли не играет.

**Готово, когда:** бой длится 15–45 сек, бота можно победить и можно проиграть, победа ощущается заслуженной.

### 3.2 Результат матча + рестарт `[Sonnet]`
Экран результата: победа/поражение, счёт, время боя, кнопка «ещё раз» (полный ресет петли без перезагрузки сцены).

**Готово, когда:** можно играть матчи подряд 10 минут без утечек и багов ресета.

### 3.3 🔴 ПЛЕЙТЕСТ-ЧЕКПОИНТ — руками, без ИИ
Играть самому и дать поиграть 3–5 людям. Вопросы:
1. Интересно ли жать 60 секунд фарма? Или это «загрузка перед боем»?
2. Бой длится >10 сек? Можно ли отыграться, отстав в фарме?
3. Хочется ли нажать «ещё раз»?

**Если «нет» по п.1** → добавить взаимодействие через стену (бойницы, отправка волны противнику, спорный бонус на стене) и повторить тест. **Если «нет» по п.3** → чинить ядро, НЕ идти дальше. Карточки и сеть не спасают скучное ядро.

## Этап 4 — Сеть (только после успешного чекпоинта)

### 4.1 Архитектура сетевого матча `[Fable]`
Дизайн-документ + скелет кода: host-authoritative NGO поверх WS-транспорта. Что синхронизируется: состояния MatchController, позиции/выстрелы/HP танков, счёт. Фарм-фаза почти не требует синхрона (каждый на своей половине, чужие фарм-боты не реплицируются — только счёт и силуэт активности). `NetworkInputSource` как третья реализация ввода.
Здесь же решить: реконнект (короткий грейс-период), выход игрока (замена ботом), кто хост.

**Готово, когда:** документ в репе + компилирующийся скелет. Ревью документа Фейблом отдельной сессией.

### 4.2 Мастер-сервер: матчмейкинг `[Fable]`
На VPS, по образцу серверной части World-of-Mine. Функции: очередь поиска, составление пары, обмен адресами/токенами, таймаут → «играй с ботом». REST или WS — как в WoM, переиспользовать максимум готового.

**Готово, когда:** два клиента (редактор + билд) находят друг друга и попадают в один матч; одиночный клиент через N сек получает бота.

### 4.3 Синхронизация боя `[Opus]`
Реализация по документу 4.1: репликация танков, снарядов, урона, состояний матча. Интерполяция движения оппонента (WebSocket = TCP, будут микрозадержки — сглаживать обязательно).

**Готово, когда:** матч играется по сети между браузером и редактором, рассинхронов счёта/HP нет, при 200мс пинга дуэль остаётся играбельной.

### 4.4 Обрывы и подмена ботом `[Opus]`
Дисконнект соперника в фарме → бот тихо доигрывает его билд. В дуэли → бот перехватывает управление с текущим состоянием. Свой дисконнект → экран «связь потеряна», очки сессии не сгорают.

**Готово, когда:** убийство процесса одного клиента в любой фазе не ломает матч второму.

## Этап 5 — Мета: карточки (первая порция)

### 5.1 Система карточек — данные `[Fable]`
`CardDefinition` (ScriptableObject): эффект на `TankStats`/поведение, редкость, уровни. Слоты: 3 карточки в билде. Влияние карточек — через `TankBuild.ComputeStats()`, никаких if-ов по конкретным карточкам в геймплейном коде. Комбинирование — заложить интерфейс, НЕ реализовывать.

**Готово, когда:** 6–8 простых карточек (числовые модификаторы), выбор 3 перед матчем, статы применяются.

### 5.2 Награды и сохранение `[Sonnet]`
Карточка после матча (пул наград по результату), сохранение профиля: локально + абстракция `ISaveProvider` (потом за ней Yandex-сейвы). Улучшение дубликатами.

**Готово, когда:** прогресс переживает перезапуск, за 10 матчей собирается ощутимая коллекция.

### 5.3 Дроп перков с фарм-ботов `[Sonnet]`
Рандомные перки на время матча (не навсегда) с редких фарм-ботов. Немного — 4–6 штук, только числовые.

**Готово, когда:** перк подбирается, действует до конца матча, виден в HUD.

## Этап 6 — Площадки и релиз

### 6.1 Вес и загрузка Web-билда `[Opus]`
Бюджет: **≤ 20 МБ Brotli**. Аудит билда (Build Report), сжатие текстур/аудио, кастомный лоадер с прогресс-баром. Проверка на реальном мобильном браузере (Unity 6 их поддерживает официально — но проверять всё равно руками).
Здесь же — опционально попробовать **WebGPU-бэкенд** Unity 6 и замерить: если даёт заметный прирост, оставить как дополнительный вариант с фолбэком на WebGL 2. Если экспериментальность вылезает багами — не тратить время.

**Готово, когда:** холодная загрузка на среднем телефоне ≤ 15 сек, билд ≤ 20 МБ.

### 6.2 Yandex Games / Playhop SDK `[Opus]`
Переиспользовать YG Plugin-опыт из WoM: авторизация, сейвы в облако (за `ISaveProvider` из 5.2), реклама (interstitial между матчами, rewarded за бонус-награду), метрики.

**Готово, когда:** проходит модерацию площадки в черновике, реклама показывается в нужных местах и не во время боя.

### 6.3 Android-порт `[Sonnet]`
Тач-управление (два стика), новый keystore (**НЕ в репозитории**, включить Play App Signing), адаптация UI под тачи.

**Готово, когда:** играбельно на среднем Android, собирается aab.

### 6.4 Софт-запуск
Выложить на одну площадку (Playhop как менее конкурентную), 1–2 недели смотреть метрики: удержание D1, длина сессии, доигрывают ли матч до конца. Итерации по данным.
